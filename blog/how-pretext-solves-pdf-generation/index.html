How Pretext Solves the Hardest Problem in PDF Generation

A technical deep-dive into Pretext's two-phase architecture: how it measures glyphs, segments Unicode text, and produces optimal line breaks in pure JavaScript.

The Problem, Restated

Generating a PDF sounds straightforward: place text on a page. But the moment you need to wrap text to fit a column, you hit a wall. To know where a line should break, you need the exact pixel width of every glyph in the string at the current font size. You need to account for kerning pairs, ligatures, and the fact that a single "character" on screen might be composed of multiple Unicode codepoints. Get any of this wrong and your text overflows, overlaps, or breaks mid-word in the wrong place.

Browsers solve this with enormously complex text shaping engines (HarfBuzz, CoreText, DirectWrite). But in a pure JavaScript environment — a Node.js server generating invoices at scale, or a browser tab building a report — you do not have access to those engines. You are on your own.

This is the problem Pretext was built to solve.

How Fonts Actually Work

A font file is not a collection of images. It is a program. An OpenType font contains vector outlines for each glyph, but it also contains tables of structured data that describe how those glyphs relate to one another:

  • cmap — maps Unicode codepoints to glyph IDs inside the font.
  • hmtx — stores the advance width of each glyph: how far the cursor moves after drawing it.
  • kern / GPOS — kerning tables that adjust spacing between specific glyph pairs (e.g., "AV" is drawn tighter than "AX").
  • GSUB — substitution rules that replace sequences of glyphs with ligatures (e.g., "f" + "i" becomes the "fi" ligature).

To measure the width of a string, you must parse these tables, resolve each character to a glyph ID, look up its advance width, apply kerning adjustments, and scale everything by the font size. Pretext does all of this by reading the raw font binary — no native dependencies, no system font fallback, just bytes and arithmetic.

Unicode: Where string.length Lies to You

JavaScript's string.length counts UTF-16 code units. This is almost never what you want when measuring text for layout. Consider:

Emoji with ZWJ Sequences

The "family" emoji is constructed by joining multiple emoji with Zero-Width Joiners:

const family = '\u{1F468}\u200D\u{1F469}\u200D\u{1F467}\u200D\u{1F466}';
console.log(family);           // one visible character
console.log(family.length);    // 11 (UTF-16 code units)
console.log([...family].length); // 7 (codepoints)

One visible character. Eleven code units. Seven codepoints. For text layout, you need to treat this as a single indivisible unit — a grapheme cluster — and measure the width of the single glyph the font provides for it.

Combining Diacriticals

The letter "e" with an acute accent can be represented two ways: as the single codepoint U+00E9, or as "e" (U+0065) followed by a combining acute accent (U+0301). Both render identically, but .length returns 1 for the first and 2 for the second. A naive measurement function that iterates by .length would double-count the width.

Ligatures

In many serif fonts, the sequence "fi" is replaced by a single ligature glyph. The advance width of the ligature is not the sum of the individual "f" and "i" widths. If your measurement code does not apply GSUB substitutions, the calculated width will be wrong.

Pretext handles all of these cases. It performs proper Unicode segmentation following UAX #29 (grapheme cluster boundaries) and UAX #14 (line break opportunities), so the text is broken into units that are safe to measure and safe to break across lines.

The Pretext Architecture

Pretext uses a deliberate two-phase design that separates measurement from layout. This separation is the key insight that makes it both fast and flexible.

Phase 1: prepareWithSegments()

The first phase takes your text and font, segments the text according to Unicode line-break rules, and measures each segment:

import { prepareWithSegments } from 'pretext';
import { readFileSync } from 'fs';

const fontData = readFileSync('./fonts/Inter-Regular.otf');

const segments = prepareWithSegments(
  'The quick brown fox jumps over the lazy dog.',
  fontData,
  { fontSize: 12 }
);

// segments is an array of measured chunks:
// [
//   { text: 'The ',  width: 22.14, breakable: true },
//   { text: 'quick ', width: 30.72, breakable: true },
//   { text: 'brown ', width: 33.48, breakable: true },
//   ...
// ]

Each segment carries its exact measured width and a flag indicating whether a line break is permitted after it. The measurement accounts for kerning, ligatures, and grapheme clusters. The segmentation follows UAX #14, which defines where lines may break in Unicode text — after spaces, after hyphens, between CJK characters, and so on.

Phase 2: layoutWithLines()

The second phase takes the measured segments and a target width, and computes optimal line breaks:

import { layoutWithLines } from 'pretext';

const lines = layoutWithLines(segments, {
  maxWidth: 200,
  lineHeight: 16,
});

// lines is an array of positioned text runs:
// [
//   { text: 'The quick brown fox', x: 0, y: 0,   width: 118.8 },
//   { text: 'jumps over the lazy',  x: 0, y: 16,  width: 112.4 },
//   { text: 'dog.',                 x: 0, y: 32,  width: 22.9  },
// ]

The line-breaking algorithm minimizes raggedness across all lines, similar in spirit to the Knuth-Plass algorithm used by TeX. It does not simply fill each line greedily; it considers the paragraph as a whole to produce more even line lengths.

Why Two Phases?

The two-phase design is not just an architectural nicety. It solves a real performance problem. Measurement is expensive — it requires parsing font tables and performing glyph lookups. Layout is cheap — it is arithmetic over pre-computed widths. By separating the two, you can:

  • Re-layout at different widths without re-measuring. If a table column changes width during auto-sizing, you only run the layout phase again. The segments are unchanged.
  • Cache segments across renders. If the same text appears in multiple documents with the same font, you measure it once.
  • Parallelize measurement. In a server environment, you can measure all text blocks concurrently, then lay them out sequentially when final dimensions are known.

Integration with pdf-lib

The coordinates returned by layoutWithLines() map directly onto pdf-lib's drawing API. There is no coordinate transformation or unit conversion needed:

import { PDFDocument, rgb } from 'pdf-lib';

const pdfDoc = await PDFDocument.create();
const page = pdfDoc.addPage([595, 842]); // A4
const font = await pdfDoc.embedFont(fontData);

const originX = 50;
const originY = 800;

for (const line of lines) {
  page.drawText(line.text, {
    x: originX + line.x,
    y: originY - line.y,
    size: 12,
    font,
    color: rgb(0, 0, 0),
  });
}

Because Pretext uses the same font binary that pdf-lib embeds, the measured widths match exactly. There is no drift between what Pretext calculates and what the PDF renderer draws. This is a guarantee you cannot get from browser-based measurement approaches like Canvas measureText(), which use system fonts that may differ from the embedded PDF font.

Real-World Results

TangentFlow uses Pretext internally to handle some of the most challenging PDF layout scenarios our users encounter:

  • Long descriptions in table cells. Product catalogs often have cells with hundreds of words. Pretext measures the text, then TangentFlow iteratively adjusts column widths and re-runs layout until the table fits the page — without re-measuring a single glyph.
  • Multilingual content. Invoices that mix English, Arabic, and Chinese text. Pretext's Unicode segmentation handles the different line-breaking rules for each script automatically. CJK text breaks between any two characters; Arabic text respects cursive joining.
  • Auto-sized columns. Pretext's two-phase design makes auto-sizing practical. Measure all cell contents once, then binary-search for optimal column widths by running layout at different widths. What would be quadratic with a combined measure-and-layout approach becomes linear.

Conclusion

Accurate text layout in PDFs has historically required either a full browser engine or deep expertise in font parsing and Unicode. Pretext collapses that complexity into two function calls. It reads font binaries directly, handles the full spectrum of Unicode edge cases, and produces coordinates you can hand straight to pdf-lib.
