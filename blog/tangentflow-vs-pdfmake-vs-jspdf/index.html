TangentFlow vs pdfmake vs jsPDF vs Puppeteer: A Comprehensive Comparison

Four approaches to JavaScript PDF generation. We compare accuracy, performance, bundle size, and developer experience to help you choose the right tool.

The Landscape of JavaScript PDF Generation

Generating PDFs from JavaScript has been a persistent challenge for web developers. Whether you need invoices, reports, contracts, or certificates, you eventually face the same question: which library should I use?

The ecosystem has converged on four main approaches, each with a fundamentally different architecture and set of trade-offs:

  • jsPDF — an imperative, canvas-like API for drawing text and shapes directly onto PDF coordinates.
  • pdfmake — a declarative document-definition model that handles layout for you.
  • Puppeteer — a headless browser approach that converts HTML/CSS into PDF on the server.
  • TangentFlow — a composable pipeline using Pretext for precise text measurement and pdf-lib for PDF output.

This post breaks down the strengths and weaknesses of each approach, with code examples and a decision framework to help you choose.

jsPDF: The Pioneer

jsPDF is the oldest and most downloaded JavaScript PDF library, with over 3 million weekly npm downloads. It provides a low-level, imperative API modeled after the HTML Canvas: you position every element by hand using x/y coordinates.

Strengths

  • Tiny bundle size (~90 KB gzipped) and zero server dependencies.
  • Simple mental model — if you know Canvas, you know jsPDF.
  • Fast generation for simple, single-page documents.
  • Large community and ecosystem of plugins like jspdf-autotable.

Weaknesses

  • No built-in text wrapping or paragraph layout. You must split strings manually.
  • No native Unicode or CJK support — only standard 14 PDF fonts unless you embed a TTF.
  • Multi-page documents require manual page-break logic.
  • Coordinates are absolute, making responsive or template-driven layouts painful.

Code Example

import { jsPDF } from "jspdf";

const doc = new jsPDF();
doc.setFontSize(18);
doc.text("Invoice #1042", 20, 20);

doc.setFontSize(12);
// Manual wrapping — you handle line breaks yourself
const lines = doc.splitTextToSize(
  "Thank you for your purchase. Please remit payment within 30 days.",
  170
);
doc.text(lines, 20, 35);

doc.save("invoice.pdf");

Notice that even basic text wrapping requires an explicit call to splitTextToSize. For anything more complex — tables, headers, footers, page numbers — you are on your own or relying on third-party plugins.

pdfmake: Declarative but Heavy

pdfmake takes a fundamentally different approach. Instead of imperative drawing commands, you define your document as a JSON object, and pdfmake handles layout, pagination, and text wrapping for you.

Strengths

  • Declarative API is more readable and maintainable than coordinate-based drawing.
  • Built-in support for tables, columns, lists, headers, and footers.
  • Automatic page breaks and pagination.
  • Works entirely client-side.

Weaknesses

  • Bundle size is roughly 2 MB gzipped because it ships its own font subsetting and layout engine.
  • CJK and complex-script support is unreliable. Font embedding for non-Latin scripts requires manual configuration and often produces incorrect glyph positioning.
  • Limited control over fine typographic details — kerning, ligatures, and line-break quality are not configurable.
  • The document-definition format, while powerful, becomes unwieldy for deeply nested or conditional layouts.

Code Example

import pdfMake from "pdfmake/build/pdfmake";
import pdfFonts from "pdfmake/build/vfs_fonts";
pdfMake.vfs = pdfFonts.pdfMake.vfs;

const docDefinition = {
  content: [
    { text: "Invoice #1042", style: "header" },
    {
      text: "Thank you for your purchase. Please remit payment within 30 days.",
      margin: [0, 10, 0, 20],
    },
    {
      table: {
        headerRows: 1,
        widths: ["*", "auto", "auto"],
        body: [
          ["Item", "Qty", "Price"],
          ["Widget A", "2", "$50.00"],
          ["Widget B", "1", "$30.00"],
        ],
      },
    },
  ],
  styles: {
    header: { fontSize: 18, bold: true },
  },
};

pdfMake.createPdf(docDefinition).download("invoice.pdf");

The declarative model is appealing, but the 2 MB payload is a dealbreaker for many client-side applications. And once you need pixel-precise control over layout — say, matching a branded template exactly — you start fighting the abstraction.

Puppeteer: The Server-Side Heavyweight

Puppeteer (and its lighter cousin Playwright) takes an entirely different approach: render a full HTML page in a headless Chrome instance and call page.pdf(). This gives you the full power of CSS for layout.

Strengths

  • Pixel-perfect rendering with full CSS support — Flexbox, Grid, custom fonts, SVG.
  • If you can build it as a web page, you can export it as a PDF.
  • Full Unicode and complex-script support out of the box.
  • Great for teams that already have HTML templates.

Weaknesses

  • Requires a server with a headless browser — typically 150+ MB of RAM per instance.
  • Slow: 2–5 seconds per document is typical, making it unsuitable for real-time or interactive use.
  • Cannot run in the browser. Client-side PDF generation is not possible.
  • Docker configuration and Chrome sandboxing add significant DevOps complexity.
  • CSS print media support, while improving, still has edge cases around page breaks, orphans, and widows.

Code Example

import puppeteer from "puppeteer";

const browser = await puppeteer.launch();
const page = await browser.newPage();

await page.setContent(`
  <html>
  <style>
    body { font-family: "Helvetica", sans-serif; padding: 40px; }
    h1 { font-size: 24px; }
    table { width: 100%; border-collapse: collapse; }
    td, th { border: 1px solid #ccc; padding: 8px; }
  </style>
  <body>
    <h1>Invoice #1042</h1>
    <p>Thank you for your purchase.</p>
    <table>
      <tr><th>Item</th><th>Qty</th><th>Price</th></tr>
      <tr><td>Widget A</td><td>2</td><td>$50.00</td></tr>
    </table>
  </body>
  </html>
`);

await page.pdf({ path: "invoice.pdf", format: "A4" });
await browser.close();

For server-side batch processing where speed is not critical and you need CSS-level layout fidelity, Puppeteer is hard to beat. But the infrastructure overhead makes it a poor fit for client-side applications or high-throughput pipelines.

TangentFlow: Accurate, Lightweight, Client-Side

TangentFlow takes a composable approach that separates two concerns most libraries conflate: text measurement and PDF output. It uses Pretext — a purpose-built text layout engine — for precise glyph measurement, line breaking, and paragraph shaping. It then hands the measured layout to pdf-lib for efficient PDF construction.

Architecture

The key insight is that accurate PDF generation is fundamentally a text-measurement problem. Most PDF libraries get layout wrong because they estimate glyph widths rather than measuring them from the actual font tables. Pretext reads OpenType font metrics directly, producing line breaks and paragraph layouts that match what a professional typesetting engine would produce.

Strengths

  • Typographically accurate text layout — real font metrics, proper kerning, correct line breaking.
  • Full Unicode support including CJK, Arabic, and Devanagari scripts.
  • Runs entirely client-side with a ~160 KB gzipped footprint.
  • Composable pipeline: you can use Pretext for measurement and swap in any PDF backend.
  • Fast — typical documents generate in 50–200 ms.
  • Template-driven: define reusable document layouts that adapt to dynamic data.

Code Example

import { Pretext } from "@tangentflow/pretext";
import { PDFDocument, rgb } from "pdf-lib";

// 1. Load font and measure text with Pretext
const pretext = new Pretext();
const font = await pretext.loadFont("/fonts/Inter-Regular.otf");

const layout = pretext.layout({
  text: "Thank you for your purchase. Please remit payment within 30 days.",
  font,
  fontSize: 12,
  width: 500,       // available width in points
  lineHeight: 1.4,
});

// 2. Create PDF with pdf-lib using measured positions
const pdfDoc = await PDFDocument.create();
const page = pdfDoc.addPage([595, 842]); // A4
const pdfFont = await pdfDoc.embedFont(font.buffer);

page.drawText("Invoice #1042", {
  x: 50, y: 792, size: 18, font: pdfFont,
});

// Render each line at its measured position
for (const line of layout.lines) {
  page.drawText(line.text, {
    x: 50, y: 750 - line.y, size: 12, font: pdfFont,
  });
}

const bytes = await pdfDoc.save();
download(bytes, "invoice.pdf");

The two-step approach — measure then render — may look like more code at first glance. But it gives you something no other client-side library offers: you know exactly where every glyph will land before you write it to the PDF. This makes it straightforward to build features like balanced columns, widow/orphan control, and precise template alignment.

When to Use Each

Choose jsPDF when:

  • You need a tiny bundle and your PDFs are simple (labels, barcodes, single-line text).
  • You do not need text wrapping, Unicode, or complex layouts.

Choose pdfmake when:

  • You want a declarative API and your documents use only Latin-script fonts.
  • Bundle size is not a constraint (server-side Node.js or internal tools).
  • You need quick-and-dirty tables without writing layout code.

Choose Puppeteer when:

  • You already have HTML templates and a server environment.
  • Latency of 2–5 seconds per document is acceptable.
  • You need full CSS support including Flexbox, Grid, and web fonts.

Choose TangentFlow when:

  • You need typographically accurate PDFs generated client-side.
  • Your documents include multilingual content or non-Latin scripts.
  • Bundle size and generation speed both matter.
  • You want a composable architecture that separates measurement from rendering.
  • You are building a product where PDF quality reflects your brand.

Conclusion

Each of these tools solves a real problem. jsPDF is unmatched for simplicity. pdfmake offers the easiest path to structured documents. Puppeteer gives you the full power of CSS. But none of them solve the fundamental problem of accurate client-side text layout without significant trade-offs in size, speed, or fidelity.

TangentFlow occupies a unique position in this landscape: it delivers the typographic accuracy of a server-side approach with the speed and portability of a client-side library. By building on Pretext for measurement and pdf-lib for output, it avoids reinventing the wheel while giving developers precise control over every aspect of document layout.

If you are building a product where PDF quality matters — invoices that must match a brand template, reports with multilingual content, or documents that need to look right in every language — TangentFlow is worth evaluating.
