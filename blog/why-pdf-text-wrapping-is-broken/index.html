Why PDF Text Wrapping Is Still Broken in 2026

Most JavaScript PDF libraries still can't wrap text correctly. Here's why the problem is fundamentally harder than it looks, and what it takes to solve it properly.

The Invoice That Broke Everything

You have probably been here before. You are building a PDF invoice generator for your SaaS app. The line items table has a "Description" column that is 200 points wide. Everything looks fine during development because your test data is short: "Widget A," "Monthly subscription," "Setup fee." Then a real customer enters a product description like "Professional consulting services for Q1 infrastructure migration and cloud deployment (includes 40 hours on-site support)" and the text blows straight through the column boundary, overlapping the unit price.

You search for a fix. The library's documentation tells you to call drawText() with an x,y coordinate. There is no mention of wrapping, no mention of column constraints, no mention of overflow. You realize with a sinking feeling that the library expects you to figure out where to break the text.

This is not a bug in one library. It is a fundamental gap in nearly every JavaScript PDF generation tool available today.

Why PDF Has No Layout Engine

The root cause is architectural. PDF is a fixed-layout format. Unlike HTML, where a browser's layout engine flows text into containers, wraps at word boundaries, and reflows on resize, a PDF file contains precise instructions: draw this glyph at coordinate (72, 680), then draw this glyph at coordinate (78.4, 680), and so on. There is no concept of a "paragraph" or a "column" at the file format level.

When you call a function like this:

const doc = new jsPDF();
doc.text("Professional consulting services for Q1 infrastructure migration and cloud deployment (includes 40 hours on-site support)", 20, 30);
doc.save("invoice.pdf");

The library writes every character on a single line starting at (20, 30). If the string is longer than the page width, the text simply runs off the edge. No wrapping. No error. Just a silently broken document.

To get wrapping, you have to do it yourself:

// Manual wrapping workaround in jsPDF
const maxWidth = 180;
const lineHeight = 7;
const lines = doc.splitTextToSize(text, maxWidth);
lines.forEach((line, i) => {
  doc.text(line, 20, 30 + (i * lineHeight));
});

This looks reasonable until you test it with anything beyond basic Latin characters. The measurement engine behind splitTextToSize is where things fall apart.

How Current Libraries Fail

jsPDF: Approximate Character Widths

jsPDF measures text width using built-in character width tables for the 14 standard PDF fonts (Helvetica, Times, Courier, and their variants). These tables map each character code to a width value. The approach has two critical problems.

First, the tables only cover the Latin-1 character set. If your text contains characters outside that range—Chinese, Japanese, Korean, Arabic, Devanagari, or even accented characters from Central European languages—the measurement returns incorrect values or falls back to a default width. The text wraps at the wrong position.

Second, even for supported characters, the tables ignore kerning. The pair "AV" is narrower than the sum of "A" and "V" measured independently, because a well-designed font tucks the V under the A's overhang. jsPDF does not account for this. Over a long line of text, these small errors accumulate and the wrap point drifts.

pdfmake: Better, But Still Simplified

pdfmake improves on jsPDF by parsing actual font files (via pdfkit under the hood) and extracting glyph metrics. This gives accurate measurements for the fonts you embed. However, the line-breaking logic is simplified. It splits on whitespace and does not implement the Unicode Line Breaking Algorithm (UAX #14), which means:

  • CJK text, which has no spaces between words, does not wrap correctly. This has been an open issue for years.
  • Languages that use zero-width spaces or other non-obvious break opportunities (Thai, Khmer) are treated as a single unbreakable token.
  • Hyphenation is not supported at all.

If your users write in English only, pdfmake works. The moment your product goes international, the wrapping breaks.

Puppeteer / wkhtmltopdf: Accurate but Heavy

The workaround many teams reach for is to skip JavaScript PDF libraries entirely. Instead, they render HTML in a headless browser (Puppeteer with Chromium, or wkhtmltopdf with WebKit) and print to PDF. The browser's layout engine handles wrapping perfectly because that is what it was built to do.

The trade-offs are severe:

  • You need a server. Chromium does not run in the browser or in a serverless function without significant effort.
  • Startup time for a headless browser is measured in seconds, not milliseconds.
  • Memory usage per render is 100-300 MB. At scale, this is expensive.
  • You lose fine-grained control over the PDF output—page breaks, headers, footers, and table splitting all require CSS hacks.

"We switched to Puppeteer for PDF generation and our AWS bill tripled. Each invoice took 3-4 seconds to render. We were spinning up Lambda instances with 1 GB of memory just to generate a two-page document."

The browser approach trades one set of problems for another. You get correct text layout but lose performance, cost efficiency, and client-side capability.

What Correct Text Wrapping Actually Requires

Text wrapping is not "split the string at spaces and check if it fits." Done correctly, it requires four things that most libraries skip:

1. Glyph-Level Measurement from Real Font Metrics

You cannot measure text width without reading the actual font file. Each glyph in a font has a specific advance width stored in the hmtx table of the TrueType or OpenType file. Using approximations or default widths produces errors that compound across a line.

2. Unicode Grapheme Cluster Segmentation

A "character" on screen is not always a single Unicode code point. The flag emoji for a country is two code points (regional indicator symbols). An accented letter can be a base character plus a combining mark. A Devanagari syllable can be three or four code points rendered as a single visual unit. You must segment text into grapheme clusters (UAX #29), not code points, before measuring.

3. The Unicode Line Breaking Algorithm (UAX #14)

Unicode Technical Report #14 defines where line breaks are allowed, required, or prohibited in text. It covers thousands of cases: CJK characters can break between any two ideographs; a closing parenthesis cannot start a new line; a non-breaking space prohibits a break; a soft hyphen permits one. Implementing this correctly is a significant undertaking. Ignoring it means your wrapping is wrong for most of the world's languages.

4. Kerning Pair Adjustments

After measuring individual glyph widths, you must apply kerning adjustments from the font's kern table or GPOS table. Without kerning, measured line widths are slightly too wide, and lines that could fit in the available space are broken prematurely.

How Pretext Approaches This Differently

Pretext, the layout engine behind TangentFlow, takes a different path. Instead of reimplementing font measurement in pure JavaScript (which means parsing binary font tables and replicating years of OpenType engineering), Pretext delegates measurement to the platform that already does it correctly: the browser's font engine.

When Pretext needs to measure a run of text, it uses the browser's Canvas text measurement API with the actual font loaded in the document. The browser's text shaper handles glyph substitution, ligatures, kerning, and complex script shaping (Arabic joining, Devanagari conjuncts, CJK composition) because it already has to do all of this for rendering web pages.

On top of accurate measurement, Pretext implements a proper line-breaking algorithm that respects Unicode break opportunities. The result is mathematically precise text wrapping that works with any language, any font, and any Unicode content—all running client-side in the browser with no headless Chrome, no server, and no Lambda functions.

The Problem Is Worth Solving

Text wrapping in PDFs has been broken for so long that most developers treat it as a fact of life. They hard-code column widths, truncate long strings, or spin up headless browsers and accept the cost. None of these are real solutions.

If you are building a product that generates PDFs—invoices, reports, contracts, labels, certificates—your users will eventually put text in a field that does not fit. When that happens, the output should wrap correctly. Not approximately. Not only in English. Correctly.
